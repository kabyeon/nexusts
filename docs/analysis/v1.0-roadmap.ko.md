# NexusTS v1.0 로드맵 — 필요한 기능과 조건

> **작성일**: 2026-06-27 | **현재 버전**: v0.9.13

---

## 1. 개요

NexusTS는 현재 **v0.9.x** 단계로, v1.0에 도달하려면 안정적인 퍼블릭 API, 생태계 성숙도, 그리고 NestJS/AdonisJS가 갖춘 핵심 기능들을 따라잡아야 한다. 아래는 v1.0 이전에 반드시 해결되어야 할 항목들을 우선순위별로 정리한 문서다.

---

## 2. 필수 조건 (P0 — v1.0 차단 항목)

> 이 항목들이 해결되지 않으면 v1.0을 선언할 수 없다.

### 2.1 Authorization 모듈 (Bouncer 수준)

AdonisJS의 `@adonisjs/bouncer`, NestJS의 `@nestjs/casl` / Guards에 대응하는 **자체 권한 부여 모듈**이 필요하다.

**필요한 API:**

```ts
// Policy 기반
@Injectable()
export class PostPolicy {
  update(user: User, post: Post): boolean {
    return user.id === post.userId;
  }
}

// Gate/Ability 기반
@Injectable()
export class CommunityService {
  @Inject(AuthorizationService) declare auth: AuthorizationService;

  deletePost(ctx: Context, post: Post) {
    // 현재 사용자가 이 게시물을 삭제할 권한이 있는가?
    this.auth.authorize(ctx, 'delete', post);
    // 또는 Policy 클래스로:
    this.auth.authorize(ctx, PostPolicy, 'update', post);
  }
}
```

**참고:** Guards(`@UseGuards`)는 이미 존재하지만, Policy/Ability 레이어가 없어 도메인 로직 수준의 세분화된 권한 제어가 불가능하다.

### 2.2 Social Auth / OAuth 모듈 (Ally 수준)

AdonisJS의 `@adonisjs/ally`처럼 **GitHub, Google, Twitter, Kakao, Naver** 등 주요 OAuth 제공자를 바로 사용할 수 있는 모듈이 필요하다. better-auth가 일부를 커버하지만, 전용 Ally 스타일의 선언적 API가 부족하다.

**필요한 API:**

```ts
// nx.config.ts
export default {
  ally: {
    github: { clientId: env('GITHUB_CLIENT_ID'), clientSecret: env('GITHUB_CLIENT_SECRET') },
    google: { clientId: env('GOOGLE_CLIENT_ID'), clientSecret: env('GOOGLE_CLIENT_SECRET') },
  },
};

// Controller
@Controller('/auth')
export class AuthController {
  @Inject(AllyService) declare ally: AllyService;

  @Get('/github/redirect')
  redirect(ctx: Context) {
    return this.ally.use('github').redirect(ctx);
  }

  @Get('/github/callback')
  async callback(ctx: Context) {
    const user = await this.ally.use('github').user(ctx);
    return { id: user.id, name: user.name, email: user.email };
  }
}
```

### 2.3 Vite 통합

프론트엔드(특히 Inertia.js)와의 통합을 위해 **Vite 개발 서버**를 연결하는 공식 모듈이 필요하다. AdonisJS의 `@adonisjs/vite`에 대응.

**필요한 기능:**

- 개발 모드에서 Vite dev server URL로 자동 연결
- 프로덕션에서 `vite build` 결과물(manifest.json 기반)으로 정적 에셋 서빙
- `@vite('resources/js/app.js')` 같은 템플릿 헬퍼
- HMR(Hot Module Replacement) 지원
- React, Vue, Svelte 등 주요 프레임워크용 프리셋

### 2.4 Testing 유틸리티 모듈

현재는 **Vitest + `new Application(AppModule)`** 만 가능하다. NestJS의 `@nestjs/testing`(`Test.createTestingModule()`)이나 AdonisJS의 `@adonisjs/testing`(`httpClient()`, `loginAs()`) 수준의 전용 테스팅 모듈이 필요하다.

**필요한 API:**

```ts
import { Test } from '@nexusts/testing';

describe('UserController', () => {
  let app: Application;
  let http: HttpTestClient;

  beforeAll(async () => {
    app = await Test.createTestingModule({
      imports: [AppModule],
      overrideProviders: [
        { provide: UserService, useClass: MockUserService },
      ],
    }).compile();

    http = app.createHttpClient();
  });

  it('GET /users returns 200', async () => {
    const res = await http.get('/users');
    expect(res.status).toBe(200);
  });

  it('POST /users requires auth', async () => {
    const res = await http.post('/users').json({ name: 'test' });
    expect(res.status).toBe(401);
  });

  it('authenticated request', async () => {
    const res = await http
      .loginAs({ id: 1, role: 'admin' })
      .get('/users/1');
    expect(res.status).toBe(200);
  });
});
```

### 2.5 Debug / Inspector 툴바

개발 중 요청/응답, DI 컨테이너 상태, 데이터베이스 쿼리, 성능 프로파일링을 실시간으로 확인할 수 있는 **디버그 툴바**가 필요하다. AdonisJS의 Inspector, Laravel의 Debugbar, Django의 Debug Toolbar에 대응.

**필요한 기능:**

- 요청 목록 (method, path, status, duration)
- 데이터베이스 쿼리 로그 (실행 시간, N+1 감지)
- DI 컨테이너 resolved providers 목록
- 렌더링된 뷰/템플릿 정보
- 세션/쿠키 상태
- 메모리 사용량
- `/__nexus/debug` 같은 전용 엔드포인트
- **개발 모드에서만 활성화** (프로덕션에서 자동 비활성화)

## 3. 주요 기능 (P1 — v1.0에 포함 권장)

> 없어도 v1.0을 선언할 수 있지만, 사용자 경험에 큰 영향을 미친다.

### 3.1 세션 기반 `@CurrentUser` 데코레이터 개선

현재 `@CurrentUser()`는 better-auth에 의존적이다. 세션 미들웨어와 직접 통합되어 **better-auth 없이도** `ctx.req.user` 또는 `@CurrentUser()`로 현재 사용자를 읽을 수 있어야 한다.

```ts
@Injectable()
export class PostService {
  @Inject(Session) declare session: Session;

  // 현재 세션의 사용자 ID 읽기
  getCurrentUserId(): number | null {
    return this.session.get('userId') ?? null;
  }
}

@Controller('/posts')
export class PostController {
  @Get('/')
  async index(ctx: Context) {
    // ctx.var.user 가 세션에서 자동 복원됨
    const user = ctx.var.user;
    return this.postService.findAllByUser(user.id);
  }
}
```

### 3.2 CQRS 모듈

NestJS의 `@nestjs/cqrs`에 대응하는 **Command/Query 분리 패턴** 지원.

```ts
// commands/create-user.command.ts
export class CreateUserCommand {
  constructor(public readonly email: string, public readonly name: string) {}
}

// handlers/create-user.handler.ts
@CommandHandler(CreateUserCommand)
export class CreateUserHandler implements ICommandHandler<CreateUserCommand> {
  @Inject(UserRepository) declare repo: UserRepository;

  async execute(command: CreateUserCommand) {
    return this.repo.create({ email: command.email, name: command.name });
  }
}

// queries/get-users.query.ts
@QueryHandler(GetUsersQuery)
export class GetUsersHandler implements IQueryHandler<GetUsersQuery> {
  @Inject(UserRepository) declare repo: UserRepository;
  async execute(query: GetUsersQuery) {
    return this.repo.findAll({ limit: query.limit });
  }
}

// Module
@Module({
  providers: [
    CreateUserHandler,
    GetUsersHandler,
    CommandBus,
    QueryBus,
  ],
})
export class UserModule {}
```

### 3.3 마이크로서비스 Transport (메시징)

현재는 gRPC만 지원한다. **Kafka, RabbitMQ, NATS** 중 최소 하나의 메시징 transport를 자체 모듈로 제공해야 한다.

```ts
@Injectable()
export class OrderService {
  @Inject(MicroserviceClient) declare broker: MicroserviceClient;

  async placeOrder(order: Order) {
    // Kafka/RabbitMQ 토픽으로 메시지 발행
    await this.broker.emit('order.placed', order);
  }
}

// 별도 서비스에서 소비
@Injectable()
export class NotificationService {
  @OnMicroserviceEvent('order.placed')
  async sendConfirmation(order: Order) {
    await this.mail.send({ to: order.email, subject: '주문 확인', ... });
  }
}
```

### 3.4 라우트 버전 관리

API 버전 관리를 위한 내장 지원.

```ts
// 옵션 A: 데코레이터 기반
@Controller({ path: '/users', version: '1' })
export class UserControllerV1 {}

@Controller({ path: '/users', version: '2' })
export class UserControllerV2 {}

// 옵션 B: URL 프리픽스 자동 생성
// /v1/users, /v2/users
@Module({
  controllers: [UserControllerV1, UserControllerV2],
  version: '1',
})
```

### 3.5 직렬화 (Serialization) 모듈

응답 데이터의 직렬화를 제어하는 모듈. NestJS의 `ClassSerializerInterceptor` + `class-transformer`에 대응.

```ts
// Exclude 민감한 필드
@Injectable()
export class UserService {
  @SerializeOptions({ strategy: 'excludeAll' })
  findById(id: number) {
    return this.repo.findById(id); // password, ssn 등 자동 제외
  }
}

// 또는 데코레이터 기반
class User {
  @Expose() id: number;
  @Expose() name: string;
  @Exclude() password: string;
}
```

## 4. 개선 사항 (P2 — v1.0 이후 또는 v1.0에 포함 가능)

> 있으면 좋지만 v1.0 차단 조건은 아니다.

| 항목 | 설명 | 우선순위 |
|------|------|---------|
| **VineJS 스타일 Validation DSL** | Zod 래퍼 — `schema.validate(request.all())` 형태의 AdonisJS 스타일 API | P2 |
| **Lucid 스타일 Active Record ORM** | Drizzle 위에 `User.find(1)`, `user.save()`, `user.delete()` 형태의 액티브 레코드 레이어 | P2 |
| **멀티 ORM 공식 지원** | Prisma, TypeORM 어댑터 | P2 |
| **Socket.IO 지원** | `@nexusts/ws`에 Socket.IO transport 추가 (기존 WS는 Bun-native) | P2 |
| **Content Collections (CMS)** | AdonisJS v7의 `@adonisjs/content` — Markdown 기반 콘텐츠 관리 | P2 |
| **Cache `@Cacheable` 데코레이터** | 메서드 레벨 캐싱 데코레이터 (`@nestjs/cache-manager` 스타일) | P2 |
| **컴프레션 미들웨어** | Hono의 `compress()`를 `@nexusts/compression`으로 래핑 | P2 |
| **CLI 플러그인 시스템** | `@nestjs/cli` schematics 스타일의 확장 가능한 CLI | P2 |
| **환경별 설정 (dev/staging/prod)** | `nx.config.ts`의 다중 환경 프로필 | P2 |
| **Sequelize/Mongoose 지원** | 추가 ORM 어댑터 | P3 |
| **프론트엔드 스캐폴드 템플릿** | React/Vue/Svelte 프로젝트를 함께 생성하는 `nx new` 옵션 | P3 |
| **Docker 이미지 공식 제공** | `nexusts/nexusts` 도커 이미지 | P3 |
| **성능 벤치마크 대시보드** | GitHub Pages에 공개된 지속적 벤치마크 차트 | P3 |

## 5. 비기능적 조건 (P0)

> 코드 기능 외에 v1.0 선언을 위해 필요한 조건.

| 조건 | 설명 | 현재 상태 |
|------|------|----------|
| **안정적인 API 표면** | v1.0 이후 major bump 없이 API 변경 불가. 현재 모든 퍼블릭 API를 면밀히 검토하고 `@experimental` 태그 정리 | ✅ 진행 중 (v0.9.x) |
| **LTS 정책 수립** | v1.0 이후 지원 기간, 보안 패치, 업그레이드 경로 문서화 | ❌ 미정 |
| **마이그레이션 가이드** | v0.9.x → v1.0 마이그레이션 문서 | ❌ 미작성 |
| **CHANGELOG 전체 감사** | 모든 v0.x breaking changes를 v1.0 마이그레이션 가이드에 집계 | ❌ 미완료 |
| **최소 2명의 메인테이너** | 단일 메인테이너 리스크 해소. v1.0 선언 전 최소 1명 추가 메인테이너 확보 | 🔴 **미해결** |
| **npm 패키지 게시 자동화 검증** | publish workflow가 실제 npm에 정상 배포됨을 검증 | ⚠️ 검증 필요 |
| **최소 1개의 프로덕션 사용 사례** | v1.0 선언 전 실제 프로젝트에서 검증된 사례 필요 | 🔴 **미해결** |

## 7. 예상 일정 (낙관적 추정)

| 단계 | 목표 | 예상 소요 |
|------|------|----------|
| **P0 기능 구현** | Authorization + Social Auth + Vite + Testing + Debug | 4~8주 |
| **P1 기능 구현** | CQRS + 메시징 + 버전 관리 + 직렬화 | 4~6주 |
| **API 표면 확정 및 문서화** | 모든 public API 검토, `@experimental` 정리 | 2주 |
| **LTS 정책 + 마이그레이션 가이드** | 정책 문서, 업그레이드 경로, CHANGELOG 감사 | 1주 |
| **메인테이너 확보 + 프로덕션 검증** | 추가 기여자 영입, 실제 프로젝트 적용 | 병행 (~8주) |
| **v1.0-rc.1** | 기능 완료, API 동결 | ~8~12주 |
| **v1.0 안정화** | 버그 수정, 문서 완료, 최종 검증 | ~2~4주 |
| **v1.0 정식 릴리스** | 🎉 | **2026년 9~10월 목표** |

---

> 현재의 개발 속도(3개월 만에 33개 모듈, 하루 수회 릴리스)를 고려하면 위 일정은 충분히 달성 가능하다. 그러나 **단일 메인테이너 의존성**이 가장 큰 리스크이며, v1.0 선언 전에 반드시 해결되어야 한다.
